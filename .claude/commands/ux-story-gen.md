You are generating **UX improvement user stories for Claude Pilot itself**.
This command is a dog-fooding tool for Claude Pilot developers, not a feature
shipped to end users. The output becomes the next PRD that the team implements.

## Inputs (read these in order)

1. **`PRODUCT.md`** (required) — product vision, personas, pains, principles,
   pillars, anti-goals, and success signals. This is the source of truth. Do
   not propose anything that violates §4 (Principles) or §6 (Anti-goals).

2. **`.local/prd/_feedback-log.md`** (optional, skip if missing) — history of
   phase reversions and rework reasons. Patterns here are strong hints for
   real pain.

3. **`.local/claude_pilot/state/usage-analytics.json`** (optional, skip if
   missing) — feature usage counters. Unused features are candidates for
   discovery improvement or removal, not new feature piles.

4. **Existing `.local/prd/prd-*/stories.md`** (required scan) — read titles
   only, to avoid proposing stories that are already in flight or shipped.
   Do not duplicate.

5. **Memory: `memory/feedback_ux_vision.md`** (optional) — older UX intent
   notes from the maintainer. Treat as historical context, not current
   direction. `PRODUCT.md` overrides it on any conflict.

## Process

Work in this order. Do not skip steps.

### Step 1 — Ground yourself in PRODUCT.md
Re-read §2 (Persona), §3 (Pains), §4 (Principles), §5 (UX Pillars), and §6
(Anti-goals). Keep these in working memory while generating.

### Step 2 — Gather signal
Read feedback log, usage analytics, and recent PRD titles if they exist.
Build a short list of real pain signals. Note any feature that has 0 usage
after being shipped — that's a signal too.

### Step 3 — Generate candidate stories
Draft 5 to 8 candidate user stories. Each must satisfy all of the following:

- **Traces to at least one numbered pain from §3** (cite the number)
- **Belongs to exactly one UX pillar from §5** as primary
- **Targets at least one persona from §2** (P1/P2/P3)
- **Does not violate any anti-goal from §6**
- **Is not a duplicate** of an existing PRD in `.local/prd/`
- **Has acceptance criteria** that can be evaluated by a gate rule (checklist,
  keyword, or file_exists). Avoid vague criteria like "feels better".

### Step 4 — Prioritize
Assign High / Mid / Low to each story. Justify priority in one sentence using
one of: pain severity, persona coverage, pillar gap, or North Star impact
(Autopilot completion rate — see §7).

### Step 5 — Write the PRD
Create a new directory under `.local/prd/` named `prd-ux-<YYYYMMDD>` (use the
current date). If that directory already exists today, append `-b`, `-c`, etc.

Inside, write `stories.md` using this exact structure:

```markdown
# UX Improvement Stories — <YYYY-MM-DD>

Generated by `/ux-story-gen` from PRODUCT.md.

## Summary
<2-3 sentences: which pains this batch focuses on, which pillars get most
attention, any notable signals from feedback/analytics.>

## Stories

### Story 1 — <title>
- **Persona:** P1 / P2 / P3
- **Pillar:** Visibility / Guidance / Resilience / Parallel / Curation / Learning
- **Pain:** #<N> (<short name from §3>)
- **Priority:** High / Mid / Low — <one-line justification>

**As a** <persona description>,
**I want** <capability>,
**so that** <outcome>.

**Acceptance criteria** (each line is a gate-checkable checkbox):
- [ ] <criterion 1>
- [ ] <criterion 2>
- [ ] <criterion 3>

**Notes** (optional — implementation hints, related code areas, risks)
- <note>

---

### Story 2 — ...
(repeat)
```

## Output constraints

- **Do not** write code, run tests, or modify any file outside the new PRD
  directory. This command only produces stories.
- **Do not** invent features that violate P1 (simplicity through UI
  segregation). If a story implies adding a whole new UI area, note that as
  an explicit risk.
- **Do not** propose anything that contradicts the cmux + desktop baseline
  (P4).
- **Keep stories small.** If a story cannot be implemented in a single PRD
  cycle (Requirements → Design → Implement → Review → Ship), split it.
- **Cite PRODUCT.md sections** when a story needs a judgment call. Example:
  "Per P2 (pain-driven), this story defers skill search improvements until
  we see evidence of the pain."

## When you are done

Print to stdout:
1. The path of the created `stories.md`
2. A one-line summary of each story (title + priority)
3. A note if you had to skip inputs because they were missing

Do not commit the PRD. The developer reviews it first.
